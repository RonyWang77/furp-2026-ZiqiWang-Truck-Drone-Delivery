# Step 1: Feasibility-First Improvement for E-VRPTW Reproduction  
## 1. Overview 

This week’s work continued from last week’s E-VRPTW reproduction experiments. Last week, the basic framework for calling OR-Tools, GA, PyVRP, and POMO was completed, but the results showed low feasibility. Many generated routes could cover customers, but violated time window, battery, or charging constraints.

Therefore, the focus this week shifted from “generating routes using different algorithms” to “generating feasible routes under complex constraints.” A feasibility-first repair mechanism was added, and ALNS was introduced as a new method for handling complex routing constraints.
---

## 2. Main Improvements 

The main improvements this week include:

1. A common route repair module was added to handle complex constraints.    
2. The routes generated by OR-Tools, GA, and PyVRP were connected to the repair process.  
3. ALNS was added as a new method.  

The optimization priority was adjusted as follows:

1. Minimize total distance
2. Minimize waiting and charging cost
3. Ensure zero constraint violation

In implementation, constraint violation is treated as a hard requirement. A solution is meaningful only when capacity, time window, battery, and customer coverage violations are all zero.

## 3. Feasibility-First Route Repair 

A common route repair mechanism was added to improve feasibility. Instead of directly accepting the candidate routes generated by the original algorithms, each route is repaired before evaluation.

The repair process is:

Candidate route
-> Keep customer visiting order
-> Try inserting customers into existing routes
-> Check capacity, time windows, and battery constraints
-> Insert charging stations when battery is insufficient
-> Open a new vehicle route if insertion is infeasible
-> Merge feasible routes to reduce vehicle count
-> Output final feasible routes

This repair mechanism allows different algorithms to be compared under the same feasibility rules.

## 4. Methods
**1.OR-Tools Improvement** 
Problem in the Previous Version:
In the previous version, OR-Tools mainly followed a CVRP-style model. It considered distance and capacity, but did not fully model time windows, battery constraints, or charging stations during route construction.

Improvements This Week:
OR-Tools was kept as the main route generator.
Capacity Dimension was used to control vehicle load.
Time Dimension was added so that customer time windows are considered.
The route generated by OR-Tools is passed to the common repair module.
The repair module inserts charging stations and checks battery feasibility.


Algorithm Logic:
EVRPTW instance
-> Build OR-Tools RoutingModel
-> Add distance cost
-> Add capacity constraint
-> Add time window constraint
-> Generate VRPTW-style main route
-> Apply route repair
-> Insert charging stations
-> Merge feasible routes
-> Output final solution

OR-Tools is now a stable and efficient baseline method.

**2. GA Improvement** 
Problem in the Previous Version:
The previous GA used customer permutation encoding and generated routes through crossover and mutation. However, the decoding process was simple and mainly split routes by capacity. It did not actively insert charging stations or repair time window and battery violations.

This week, the GA decoder and fitness function were improved.
The individual is still encoded as a customer permutation.
Crossover and mutation remain in the original GA style.
The decoder now calls the route repair mechanism.
The fitness function was changed to a feasibility-first objective.


The new fitness priority is:
constraint violation penalty
+ vehicle count penalty
+ distance cost
+ waiting and charging cost

Algorithm Logic:
Generate customer permutation
-> Selection, crossover, mutation
-> Decode permutation through route repair
-> Insert charging stations
-> Check capacity, time windows, and battery
-> Calculate fitness
-> Keep better individuals
-> Output best solution

GA achieved the best average vehicle count and distance in the current experiments, but required the longest runtime.

**3. PyVRP Improvement** 
Problem in the Previous Version:
PyVRP is effective for VRP, CVRP, and VRPTW, but it is not specifically designed for battery recovery, charging station insertion, and charging time in E-VRPTW.

Improvements This Week:
PyVRP is still used to generate the main VRPTW route.
Customer demand, service time, time windows, and vehicle capacity are modeled in PyVRP.
The PyVRP route is passed to the common repair module.
Charging stations and battery constraints are handled during repair.

Algorithm Logic:
EVRPTW instance
-> Convert to PyVRP Model
-> Add depot, clients, vehicle type, and edges
-> Solve VRPTW main route
-> Apply route repair
-> Insert charging stations
-> Check battery and time window feasibility
-> Output final solution

PyVRP runs efficiently, but its final route quality depends heavily on the repair process.

**4. New ALNS Method** 

ALNS was added because its destroy-and-repair structure is suitable for routing problems with complex constraints. Compared with POMO, ALNS does not require model training and is easier to adapt to time windows, battery constraints, and charging stations.

Basic Idea:
Generate initial solution
-> Destroy part of the solution
-> Repair the solution
-> Decide whether to accept the new solution
-> Repeat

## 8. Discussion 
OR-Tools
Advantages:

Fast runtime.
Stable feasible solutions.
Time windows are modeled through Time Dimension.

Disadvantages:

Battery and charging constraints are mainly handled by post-repair.
Further modeling is needed for a stricter E-VRPTW formulation.


GA
Advantages:

Best average vehicle count.
Shortest average distance.
Flexible encoding and fitness design.

Disadvantages:

Long runtime.
Sensitive to parameters and random search.

PyVRP
Advantages:

Fast and stable.
Good at VRPTW-style route generation.

Disadvages：

Charging constraints are not directly modeled.
Final solution quality depends heavily on repair.

ALNS
Advantages:

Suitable structure for complex constraints.
Destroy/repair mechanism can naturally include charging and battery logic.
More practical than POMO for this stage.

Disadvantages:

Current operators are still basic.
Route quality is not yet better than GA or OR-Tools.

## 9. Conclusion / 本周结论
This week successfully improved the feasibility of the E-VRPTW reproduction framework. By adding a feasibility-first repair mechanism and connecting it to OR-Tools, GA, PyVRP, and ALNS, all tested methods achieved 100% feasibility on R101, C101, and RC101 instances with 10, 25, and 50 customers.

The main findings are:
GA achieved the best route quality but required the longest runtime.

OR-Tools provided a strong balance between solution quality and efficiency.

PyVRP was fast but depended strongly on the repair mechanism.

ALNS is promising for future complex extensions, but its operators need further improvement.

The feasibility-first repair mechanism effectively solved the low-feasibility problem from last week.
